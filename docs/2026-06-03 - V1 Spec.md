# IndicoInk: Indico Slide Notes App — Product Specification

## 1. Purpose

The app helps a conference attendee take notes directly in the context where ideas occur: while viewing conference slides. Instead of maintaining a separate notebook organized manually by session and talk, the app organizes notes around an Indico event, its agenda, individual talks, and the specific slides that the user annotated.

The primary output is a compact Markdown conference note document containing only the talks and slides the user marked up, with links back to the original Indico contributions.

## 2. Core Concept

A user provides an Indico event URL. The app fetches the event agenda and lets the user browse the conference structure. When the user selects a talk with slides, the app displays those slides. The user can annotate slides using a pen/stylus or by adding typed text. The app remembers all annotations across sessions and across multiple days of a conference.

At any time, the user can export the annotated conference notes as a single self-contained Markdown file.

## 3. Target User

The initial target user is a technically sophisticated conference attendee who frequently attends large scientific conferences, especially conferences hosted on Indico. The user may attend many talks, only annotate a subset, and wants the exported notes to preserve the context of each annotated idea.

The app should work well for large, complex conferences such as CHEP 2026.

## 4. Primary User Goals

* Open an Indico conference by URL.
* Browse or search a large conference agenda.
* Open the slides for a selected talk.
* Mark up slides naturally using a stylus, pen, mouse, or touch input.
* Add typed text notes directly on a slide.
* Resume work later without losing annotations.
* Export a concise, self-contained Markdown document containing only annotated material.
* Preserve links back to the original Indico talks and materials.

## 5. Non-Goals for Initial Specification

The initial product specification does not yet choose a technical stack, database, UI framework, PDF renderer, synchronization model, or packaging mechanism. Those choices will be considered after the desired behavior is specified.

The initial version is also not intended to be a general-purpose note-taking app independent of Indico. Its organizing principle is the Indico event agenda and slide materials.

## 6. High-Level Workflow

### 6.1 Open Conference

1. User starts the app.
2. The app shows a table of contents or conference library containing previously opened conferences.
3. User may reopen an existing conference, enter a new Indico event URL, or launch the app with an Indico URL supplied by another program.
4. For a new conference, the app fetches the event metadata and agenda.
5. The app displays the conference structure in a navigable form.
6. The conference is remembered locally so the user can return to it later.
7. Each conference in the library has a trash/delete control. If the user deletes a conference from the library, all local data for that conference is deleted, including agenda cache, slide cache, annotations, and export state.

### 6.2 Browse Agenda

The user can navigate the agenda by day, session, contribution, and talk. The agenda view must support very large and nested Indico events. The user should be able to find a talk without manually expanding every session.

Expected agenda capabilities:

* Show event title, dates, and source URL.
* Show sessions, blocks, contributions, and talks in a readable hierarchy.
* Support large agendas without becoming sluggish.
* Allow search by talk title, speaker, session, contribution ID, or keyword.
* Clearly indicate which talks have available slide material.
* Clearly indicate which talks already have user annotations.

### 6.3 Open Talk Slides

When a user selects a talk, the app shows the talk metadata and available materials. If slides are available, the user can open them in a slide viewer.

The slide viewer should support:

* Viewing one slide in a continuous roll (like pages in a PDF document)
* Moving forward and backward through slides.
* Jumping to a slide number.
* Seeing the current slide number and total number of slides.
* Zooming and panning as needed.
* Returning to the agenda without losing state.

### 6.4 Annotate Slides

The user may annotate a slide in two main ways:

1. Freehand pen/stylus markup.
2. Typed text inserted at a chosen location on the slide.

Annotations are attached to a specific slide of a specific talk in a specific Indico event.

Expected annotation behavior:

* Pen strokes should feel natural and low-friction.
* Typed notes should be placed by clicking or tapping on the slide.
* The user should be able to edit or remove typed notes.
* The user should be able to erase or undo recent pen strokes.
* The app should visually distinguish original slide content from user annotations.
* The app should mark the talk and slide as annotated once any annotation is added.

### 6.5 Persist Work

The app must preserve all annotation state if the user quits and restarts. This is essential because conferences run over multiple days.

Persistence requirements:

* The app remembers previously opened conferences.
* The app remembers fetched agenda metadata.
* The app remembers downloaded or cached slide materials when appropriate.
* The app remembers all annotations.
* The app remembers enough talk and slide identity information to reconnect annotations to the original Indico contribution.
* The app should avoid data loss if closed unexpectedly.
* The user should be able to annotate and review slides offline if the slides were previously cached locally.

### 6.6 Export Conference Notes

At any time, the user can export the conference notes.

The export is a single self-contained Markdown file containing:

* Conference title.
* Conference URL.
* Conference Date
* Export date/time.
* One (markdown) section per annotated talk.
* A link to the original Indico contribution for each talk.
* Talk metadata when available, such as title, speaker(s), session, date, and time.
* Only the slides that the user annotated.
* Rendered slide images including the user’s annotations, embedded inline in the Markdown file using data URLs or an equivalent single-file representation.

The export should omit:

* Talks with no annotations.
* Slides with no annotations.
* Unused agenda material.

The Markdown should be suitable for later reading, sharing, conversion to PDF, or inclusion in a personal knowledge system. Sharing the export should require sending only one file.

## 7. Functional Requirements

### 7.1 Indico Event Input

* The user can start the app normally and enter an Indico event URL.
* The app can also be launched with an Indico URL supplied externally, for example from another program, command line, browser integration, or shortcut.
* The app validates that the URL appears to refer to an Indico event.
* If the app cannot fetch the event, it gives a clear error and preserves any previously stored data.
* The app should support private or access-controlled Indico events via a user-provided Indico API key.

### 7.2 Agenda Fetching

* The app fetches the event agenda from Indico.
* The app handles large and deeply nested agendas.
* The app stores enough agenda metadata locally to support later browsing even after restart.
* The app should be robust against missing speakers, missing times, incomplete material links, and inconsistent Indico metadata.
* The app should refresh the agenda data periodically or upon request. During refresh the user should be able to continue to work with the old data.
* If an agenda refresh changes, moves, renames, or removes a talk that already has annotations, the app must preserve the existing annotations and continue to include them in exports. The app must not discard or hide the user's work without explicitly asking first.
* If an agenda refresh adds new slide material to a talk that previously had no slides, the app should automatically make the material available without interrupting or explicitly notifying the user.

### 7.3 Talk and Material Handling

* The app identifies talks/contributions that have slide materials.
* The app allows the user to open slide material associated with a talk.
* In the initial version, slide PDFs are downloaded and cached only when the user explicitly opens a talk.
* The app keeps the original Indico contribution URL for each talk.
* If multiple slide files are attached to a talk, the app lets the user choose which one to open.
* If multiple PDF slide files are attached to a talk, the user's chosen deck should be remembered as the default for that talk.
* If a talk has no slides, the app should indicate that clearly.
* Only slides that are PDF format are eligible for annotation and viewing within the talk.
* If the app detects that slide files on Indico have changed after annotations already exist, the app should ask the user whether to download the new version and discard existing annotations associated with the old version.

### 7.4 Slide Display

* The app displays slides in a readable, annotatable viewer.
* The app supports slides only in PDF format.
* The app should preserve slide aspect ratio.
* The slide viewer should use a continuously scrolling document model similar to a standard PDF viewer.
* The app should support zooming for detailed annotation.
* Slide navigation should be fast enough to use during a live talk.

### 7.5 Pen Annotation

* The user can draw freehand marks on a slide.
* Version 1 annotation tools should include pen input, eraser support, and undo/redo functionality.
* Version 1 should use black pen strokes as the default pen appearance.
* Future versions may support selectable pen thickness, selectable colors, highlighting tools, and richer annotation styling.
* Pen strokes are stored permanently as editable vector overlay data, not only as a final flattened bitmap.
* Flattened annotated slide images are generated only for export or display cache purposes.

### 7.6 Typed Annotation

* The user can click or tap a slide location and add typed text.
* Typed notes are visually placed on the slide.
* Typed notes are stored permanently as editable text overlay data.
* The user can edit, move, or delete typed notes.
* Version 1 typed annotations should support only simple editable text placed directly on the slide.
* Version 1 should use black typed text as the default annotation appearance.
* More advanced text formatting and layout features may be added in future versions.

### 7.7 Conference Library and Deletion

* The app provides a conference library or table of contents listing conferences the user has previously opened or edited.
* Each listed conference shows enough information to identify it, such as title, dates, source URL, and last modified time.
* Each listed conference has a trash/delete button.
* Deleting a conference removes all local data associated with that conference, including fetched agenda metadata, cached slides, annotations, and any generated local export artifacts.
* The app should protect against accidental deletion with a confirmation step or equivalent undo mechanism.

### 7.8 Annotation State Indicators

The app should make it easy to see what has already been annotated.

Useful indicators include:

* Agenda marker showing talks with annotations.
* Talk-level marker showing number of annotated slides.
* Slide navigator marker showing which slides have annotations.

### 7.9 Export

* The user can export the current conference annotations at any time.
* Export creates a single Markdown file with all annotated slide images embedded inline using base64 `data:` image URLs.
* Each exported slide image includes the original slide plus user annotations.
* Version 1 exports should lean toward higher-quality slide images rather than minimizing Markdown file size.
* Export preserves a stable structure so repeated exports are predictable.
* Each export operation should ask the user for an output filename and location. If the selected file already exists, the app may overwrite it after normal operating-system confirmation behavior.
* If the user exports a conference with no annotated slides, the app should warn the user and produce no export output.
* Export should not require the user to manually select talks or slides; the app automatically includes only annotated content.

## 8. Data Model Concepts

The app should conceptually track the following entities:

### 8.1 Conference

* Indico event URL.
* Event title.
* Event dates.
* Local cache location or identifier.
* Last fetched time.

### 8.2 Agenda Item

* Session/block hierarchy.
* Contribution/talk identity.
* Title.
* Speaker(s), if available.
* Date and time, if available.
* Original Indico contribution URL.
* Material links.

### 8.3 Slide Deck

* Source material URL.
* Local cached copy, if available.
* Format, such as PDF.
* Number of slides/pages.
* Association with talk/contribution.

### 8.4 Slide

* Deck identity.
* Slide/page number.
* Rendered view state, if needed.
* Annotation status.

### 8.5 Annotation

* Associated event, talk, deck, and slide.
* Type: pen stroke or text note.
* Position and geometry.
* Styling.
* Creation and modification time.
* Deleted or active state, if undo/history is supported.
* Editable source representation: vector stroke data for pen annotations and text object data for typed annotations.
* Export/rendered representation generated from the editable source representation when needed.

## 9. Export Markdown Structure

A representative export structure using inline embedded images:

```markdown
# Conference Notes: <Conference Title>

- Source: <Indico event URL>
- Date: <Dates of Conference>
- Exported: <date/time>

## <Talk Title>

- Speaker(s): <speakers if available>
- Session: <session if available>
- Time: <date/time if available>
- Indico: <contribution URL>

### Slide <N>

![Annotated slide <N>](data:image/png;base64,<embedded-image-data>)

### Slide <M>

![Annotated slide <M>](data:image/png;base64,<embedded-image-data>)
```

The exact Markdown format can evolve, but it should remain simple, readable, easy to process with other tools, and self-contained as a single file.

## 10. Windows Requirements

The app must run on Windows.

Windows-specific expectations:

* Installable or launchable without a complicated developer setup.
* Works with pen/stylus input on Windows tablets and 2-in-1 devices.
* Can be launched with an Indico URL from another program.
* Stores local data in an appropriate user-specific application data location.
* Does not require administrative privileges for normal use.

## 11. Reliability Requirements

* Annotation data should be saved frequently enough that accidental app closure loses little or no work.
* Export should not corrupt stored annotation data.
* If a slide deck cannot be re-downloaded, existing cached annotations and cached slide data should remain usable when possible.
* The app should tolerate network interruptions after the conference and slides have been fetched.
* Once slide material has been downloaded and cached locally, the user should be able to annotate and review slides offline.

## 12. Performance Requirements

* Large agendas should remain navigable.
* Opening a talk should not require refetching the entire conference if the agenda is already cached.
* Slide-to-slide navigation should be responsive enough for use during a live presentation.
* Export time should be reasonable for the number of annotated slides, not for the full size of the conference.

## 13. Privacy and Local Data

The initial product assumption is that annotations are local to the user’s machine.

The app should not upload notes or annotations unless a future synchronization feature is explicitly added. The user should be able to locate, back up, and delete local conference data. Deleting a conference from the conference library should remove all locally stored data associated with that conference.

## 14. Possible Future Features

These are not required for the initial version but may influence design choices:

* Full-text search across annotations.
* OCR or text extraction from slides.
* AI-generated summaries of annotated talks.
* Automatic extraction of action items or follow-up questions.
* Synchronization across devices.
* Export to PDF, OneNote, Obsidian, or other note systems.
* Support for multiple simultaneous slide decks per talk.
* Support for non-PDF slide formats.
* Tagging talks or slides.
* Session-level summary pages.
* Integration with calendar or conference schedule planning.
* Manual inclusion of a slide in the summary even if there isn't an associated annotation.
* Intelligent transfer or alignment of annotations between updated slide revisions.
* Proactive downloading and caching of all conference slide material during refresh for full offline conference usage.
* A clearly documented local backup/export of the editable annotation database.

## 15. Open Product Questions

There are currently no unresolved open product questions for the initial specification.

## 16. Initial Success Criteria

The first useful version succeeds if a user can:

1. Open a large Indico event from a URL.
2. Find a talk in the agenda.
3. Open its PDF slides.
4. Draw on at least one slide.
5. Add typed text to at least one slide.
6. Quit and restart the app without losing annotations.
7. Export a single self-contained Markdown file containing only the annotated talks and annotated slides, with links back to Indico.
