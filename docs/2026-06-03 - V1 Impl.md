# IndicoInk V1 Implementation Plan

## Execution Rules

- Complete only one checklist item at a time.
- After addressing each item, run its stated verification, mark it complete, commit the change, and move directly onto the next item.
- Each check-in must summarize the change, list the verification performed and its result, and identify any decision, limitation, or follow-up discovered.
- Prioritize work that can be internally tested first. Group UI-heavy changes so they can be validated together.
- After a significant UI update, pause and let the user test the result before continuing with the next UI step.
- For every UI/UX item, read the relevant sections of [V1 Design](./2026-06-03%20-%20V1%20Design.md) and [Design Language](./design-language.md) before implementation.
- UI/UX check-ins must state which design sections were followed and include a visual or manual interaction verification at the relevant window sizes and input methods.
- Treat [V1 Design](./2026-06-03%20-%20V1%20Design.md) as the source of truth for screen structure, workflows, responsive behavior, and exceptional states.
- Treat [Design Language](./design-language.md) as the source of truth for visual tokens, components, interaction conventions, accessibility, and wording.
- For every PDF, ink, pointer, cursor, and Electron launch item, read [Electron Pen Spike Info](./2026-06-03%20-%20Electron%20Pen%20Spike%20Info.md) and reuse its validated behavior unless a new target-device test justifies a change.
- Ink-related check-ins must report mouse, pen-tip, hardware-eraser-contact, eraser-hover, and touch behavior separately.
- When PDF rendering or pointer behavior fails, inspect and report the relevant page-status, pointer-diagnostic, SVG-class, computed-cursor, and asset-path evidence before changing code.
- If you have trouble testing something (it is inaccessible, or you are blocked), stop to ask for help - do not bypass running verifications.

## 3. Early PDF and Ink Vertical Slice

- [ ] Add deterministic one-page, multi-page, text-heavy, and image-heavy PDF test fixtures. Verify the fixtures open in an external PDF reader.
- [ ] Add a native Open PDF command through a narrow preload API. Verify canceling and selecting a local PDF both behave correctly.
- [ ] Add PDF.js renderer and worker compatibility setup before loading PDF.js. Verify renderer and worker paths work in development and the packaged app.
- [ ] Verify the built and packaged renderer resolves its HTML, JavaScript, CSS, PDF.js worker, and other assets without absolute development paths. Verify a packaged launch can render a PDF while offline.
- [ ] Render one selected PDF page with PDF.js using the spike-validated `intent: 'print'` path as the initial baseline. Verify text-heavy and image-heavy sample PDFs render visibly.
- [ ] Render all PDF pages as a continuous vertical roll. Verify scrolling through a multi-page PDF remains responsive.
- [ ] Add per-page transparent SVG overlays sized to the PDF.js viewport. Verify overlays remain aligned after window resize.
- [ ] Add normalized page-coordinate and point-time helpers with unit tests. Verify points round-trip between screen and page coordinates while preserving pressure and event time.
- [ ] Centralize pointer-tool resolution and active-tool latching for mouse, pen tip, hardware eraser contact, and touch. Verify unit tests cover `button === 5`, `buttons & 32`, and release behavior.
- [ ] Treat hardware eraser hover as pen hover unless the current Electron version proves otherwise on-device. Verify eraser contact resolves to eraser without assuming hover identity.
- [ ] Drive toolbar tool state and SVG overlay classes from the centralized resolved or latched pointer tool. Verify mouse, pen, and eraser contact never produce contradictory rendered state.
- [ ] Add pointer diagnostics showing raw `pointerType`, `button`, `buttons`, `pressure`, and `isPrimary`, plus resolved tool, latched active tool, rendered tool, SVG class, and computed cursor. Verify diagnostics update during mouse, pen-tip, eraser-hover, eraser-contact, and touch input.
- [ ] Add optional main-process `webContents` `cursor-changed` diagnostics. Verify the output can distinguish renderer/CSS state from Electron or Windows cursor behavior.
- [ ] Implement black freehand strokes for pen and explicit mouse drawing mode. Verify strokes stay on the page where they began and mouse uses a crosshair tool state.
- [ ] Apply pen pressure to rendered stroke width, initially scaling up to 50% above base width as validated in the spike. Verify pressure changes are visible on the target pen device.
- [ ] Handle touch before drawable-pointer checks and route one-finger touch to the active document scroller without drawing. Verify touch scroll and pen drawing separately on the target device.
- [ ] Implement whole-stroke erasing with the hardware eraser and toolbar eraser. Verify both paths erase the intended stroke.
- [ ] Add session-only undo and redo for stroke add and erase actions. Verify reducer unit tests and toolbar behavior.
- [ ] Add a compact Pen, Eraser, Undo, and Redo toolbar following V1 Design section 5 and Design Language sections 5-6. Verify controls work with mouse, keyboard, and pen without covering the active drawing area.
- [ ] Implement cursor presentation from centralized pointer-tool state: CSS cursor for mouse/trackpad and an in-page SVG or DOM marker for pen tip and eraser contact. Verify pen uses a dot, eraser contact uses a distinct marker, and touch shows neither.
- [ ] Verify cursor behavior on the target Windows device without relying on the native stylus cursor or CSS cursor repainting during active pen input. Record native cursor limitations and in-page marker results in `ARCHITECTURE.md`.
- [ ] Keep PDF page status visible while loading, rendering, or failing. Verify a forced PDF error produces a readable page-level state.
- [ ] Add PDF blank/gray-page diagnostics for page status, canvas size, render completion, worker setup, and built renderer asset paths. Verify each failure mode can be distinguished before changing ink code.
- [ ] Run `npm run verify` and manually test a multi-page PDF on the target Windows pen device. Verify Electron, PDF.js, pen, eraser, touch-scroll, and cursor findings are recorded in `ARCHITECTURE.md`.
- [ ] Gate: accept the target-device PDF and ink vertical slice before beginning durable annotation plumbing. Verify the acceptance result is recorded in `ARCHITECTURE.md`.
